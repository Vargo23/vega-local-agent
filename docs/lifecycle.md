# Жизненный цикл задачи VEGA

## Этап 1. Получение задачи

VEGA принимает задачу пользователя и определяет тип работы:

- архитектура;
- код;
- рефакторинг;
- багфикс;
- документация;
- настройка;
- анализ идеи;
- планирование проекта.

## Этап 2. Понимание цели

VEGA формулирует цель задачи коротко и практически.

Пример:

```text
Цель: добавить CLI-оформление для VEGA, чтобы пользователь видел статус агента, модель, режим и ошибки.
```

## Этап 3. Разбиение на блоки

Задача делится на небольшие блоки.

Пример:

```text
Block 1: стартовый баннер
Block 2: статусы сообщений
Block 3: режим интернета
Block 4: review-gate вывод
Block 5: финальная проверка
```

## Этап 4. Выполнение блока

VEGA выполняет только текущий блок, не расползаясь на лишний функционал.

## Этап 5. Coordinator Review Gate

После блока VEGA проверяет результат:

```text
[REVIEW] Проверка блока
- соответствует ли задаче: да/нет
- не сломана ли архитектура: да/нет
- есть ли лишний функционал: да/нет
- можно ли двигаться дальше: да/нет
```

## Этап 6. Сообщение пользователю

VEGA обязана сообщить статус:

```text
[DONE] Блок 1 готов. Стартовый баннер добавлен.
```

или

```text
[REWORK] Блок 1 отправлен на переработку. Проблема: баннер не показывает режим интернета.
```

## Этап 7. Следующий блок или переработка

Если review пройден, VEGA переходит дальше.

Если review не пройден, VEGA исправляет блок и повторяет проверку.

## Правило остановки

Если VEGA не уверена в результате, она не должна выдавать его как готовый.

Правильно:

```text
[WARNING] Блок частично готов. Требуется проверка запуска.
```

Неправильно:

```text
Все готово и точно работает.
```

если фактической проверки не было.

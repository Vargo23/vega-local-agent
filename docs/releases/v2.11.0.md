# VEGA v2.11.0 Release Notes

VEGA v2.11.0, **Runtime Diagnostics Evolution**, provides a unified local-only,
bounded runtime diagnostics observer without adding execution authority.

## Highlights

* Immutable allowlisted runtime reports with fixed error codes.
* Exact `/doctor` help, trace status/latest/summary, and export commands.
* Explicit atomic UTF-8 JSON export with deterministic bounded retention.
* Three trace backups by default, bounded multi-file scanning, and safe aggregate
  counts.
* Backward-compatible v2.10 JSONL trace reading.

## Security contract

Reports and traces exclude prompts, user text, conversation history, evidence,
file/patch contents, tool arguments/results, stdout/stderr, query-bearing URLs,
absolute user paths, environment data, secrets, tokens, credentials,
confirmation/session data, exception text/objects, tracebacks, callables,
handlers, and callbacks. Serialization is manual and allowlisted.

Diagnostics do not invoke tools, repeat plans, call handlers or models, perform
network access, run shell commands, or modify permission, routing, synthesis, and
request results. Invalid policy, serialization, scanning, rotation, and export
fail closed with fixed codes.

## Export and retention

Only `/doctor export` creates a report under
`logs/diagnostics/reports/doctor-YYYYMMDDTHHMMSSffffffZ.json`. The path cannot be
overridden. Writes use a same-directory temporary file, flush, `fsync`, and atomic
replace. Retention removes only exact report filenames, preserves unknown files,
and uses a hard scan cap.

## Trace evolution

`VEGA_EXECUTION_TRACE` remains the explicit opt-in. The 5 MiB active-file hard
limit is unchanged. Validated policy retains `.1`, `.2`, and `.3` by default and
caps files and records scanned. Corrupt lines and oversized files are skipped;
the active file has priority, then backups in numeric order, and later valid
records within a file have priority.

## Migration impact

No migration is required. Existing valid v2.10 records and public trace APIs
remain compatible. `config/diagnostics_policy.json` is a new tracked policy;
trace/report state remains ignored and local.

## Validation

Focused and full pytest suites, compileall, identity, policy consistency, smoke,
Release Manager, and whitespace checks form the release gate.

## Known limitations

* Tracing is opt-in.
* Reports are local-only and remote telemetry is absent.
* Locking is process-local, not interprocess-safe.
* Diagnostics do not launch Ollama to probe model installation.
* Automatic publishing is absent.

# VEGA v2.6.0 - Permissions System

VEGA v2.6.0 adds production enforcement for a fixed permission policy. The
permission vocabulary contains `allow`, `confirm`, and `deny` effects, with
`low`, `medium`, `high`, and `critical` risk labels. Missing rules and evaluator
failures deny execution, while unknown registry names retain `unknown_tool`.

## Highlights

* Exact production registry-policy alignment and complete tool classification.
* `PermissionEvaluator` enforcement in `ToolExecutor` before argument validation
  and callable execution.
* Machine-readable permission failures and a fixed production executor factory.
* One-time interactive confirmation: `y`/`yes` approves once; rejected, empty,
  unknown, interrupted, failed-callback, and non-interactive input fails closed.
* In-memory session grants for policy-approved confirm tools only.
* `/permissions`, `/permissions grants`, `/permissions revoke <tool_name>`, and
  `/permissions clear` lifecycle commands. There is no direct grant command.

Session grants are exact-tool, current-process state. They begin empty and are
never persisted in configuration, checkpoints, workflows, project memory, or
logs. Workflow confirmation and interactive tool confirmation remain separate.

## Verification

The v2.6.0 release suite passes 557 discovered unit tests (one skipped), including
46 focused permission tests and the workflow-confirmation regressions. Identity,
compilation, documentation validation, and the configured release check are also
part of final release verification.

## Known limitations

* No persistent or cross-process permission grants.
* No automatic or remote approval.
* No GUI permission management.
* Non-interactive confirmation-required actions remain blocked.
* Model-driven and context-aware tool selection is not part of v2.6; it belongs
  to the planned v2.7 scope.

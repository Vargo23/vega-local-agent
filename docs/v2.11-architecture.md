# VEGA v2.11 — Runtime Diagnostics Evolution

Target release: `v2.11.0`

Status: implemented and release-validated.

## 1. Goal

v2.11 adds one safe local diagnostics layer that reports runtime subsystem state,
exports a report only on explicit request, evolves execution-trace retention, and
provides bounded trace status and aggregates. The layer is an observer and has no
tool authority.

## 2. Non-goals

The release does not add a GUI, background monitoring, autonomous execution,
another tool path, new agent permissions, a marketplace, remote telemetry,
internet transmission of diagnostics, model calls from diagnostics, arbitrary
export paths, or automatic release publishing.

## 3. Architecture

The only supported execution path remains:

```text
contextual runtime
    -> plan execution
    -> PlanExecutor
    -> ToolExecutor
```

Diagnostics observe immutable and persisted safe state:

```text
ProductionSnapshot -----------+
model/RAG/memory/terminal -----+--> RuntimeDiagnosticsReport
execution trace store --------+          |
release/runtime file booleans -+          +--> /doctor summary
                                          +--> /doctor export (explicit only)
```

`core/runtime_diagnostics.py` owns policy validation, immutable report models,
building, serialization, formatting, atomic export, and report retention.
`core/execution_trace.py` owns trace lifecycle, serialization, rotation, and
bounded active/backup scans. `core/agent_runtime.py` only routes exact commands.

## 4. Diagnostics policy contract

`config/diagnostics_policy.json` uses exact `schema_version: 1` fields:

* `trace_store_path`;
* `max_trace_file_bytes`;
* `retained_trace_backups`;
* `max_trace_scan_files`;
* `max_trace_records`;
* `doctor_reports_dir`;
* `max_doctor_report_bytes`;
* `retained_doctor_reports`.

Unknown, missing, and duplicate fields are rejected. Boolean-as-integer, wrong
types, zero/negative values where meaningless, and values above compiled hard
caps are rejected. No partial policy is returned. The 5 MiB active trace ceiling
cannot be enlarged by configuration; default backup retention is three.

## 5. Report schema

`RuntimeDiagnosticsReport` is a frozen slotted dataclass. Its exact top-level
allowlist contains identity, UTC creation time, report type, overall status,
fixed error codes, production snapshot, model runtime, documents, memory,
terminal policy, execution traces, and runtime-file booleans.

Nested contracts contain only bounded identifiers, booleans, counts, fixed
codes, and safe aggregates. Manual `to_safe_dict()` methods construct every
security-sensitive dictionary. `dataclasses.asdict`, `vars`, `__dict__`, and
untrusted `repr` are not used for serialization.

## 6. Trace retention contract

Tracing is disabled by default and remains enabled only by
`VEGA_EXECUTION_TRACE`. Terminal traces append as UTF-8 JSONL. `started` traces
are never persisted. Rotation under the existing process-local lock is:

```text
delete oldest
.2 -> .3
.1 -> .2
active -> .1
append to new active
```

The exact retained count comes from validated policy and cannot exceed the hard
cap. Zero backups is supported when scan-file count is correspondingly one.
Directory, serialization, rename, unlink, and append failures return `None` and
cannot change execution results.

## 7. CLI contract

Exact deterministic commands are:

```text
/doctor
/doctor help
/doctor trace status
/doctor trace latest
/doctor trace summary
/doctor export
```

`/doctor` formats the common report. `trace status` shows opt-in state, relative
path, active bytes, backups, valid-record count, and corruption flag. `trace
latest` preserves the v2.10 safe summary. `trace summary` shows bounded terminal
counts and safe codes. `export` creates JSON. Extra export arguments and unknown
subcommands never create files and return fixed usage output; fuzzy routing is
not used.

## 8. Redaction contract

Traces and reports never contain prompts, system prompts, raw user text,
conversation history, evidence, file or patch contents, tool arguments/results,
stdout/stderr, query-bearing URLs, absolute user paths, environment dumps,
tokens, cookies, keys, credentials, confirmation tokens, session grants,
exception messages/objects, stack traces, arbitrary object representations,
callables, handlers, or callbacks.

## 9. Error vocabulary

Runtime boundaries expose a small fixed set including:

```text
diagnostics_policy_error
diagnostics_build_failed
diagnostics_serialization_failed
diagnostics_export_failed
diagnostics_report_too_large
diagnostics_retention_failed
trace_store_unavailable
trace_record_invalid
trace_scan_limit_reached
```

Subsystem absence uses fixed model/documents/memory/terminal codes. Production
issue identifiers come only from the fixed policy-consistency enum. Raw external
exception text is never copied to a report or doctor output.

## 10. Path confinement

Policy accepts only normalized relative paths. Absolute paths, drive-qualified
paths, parent traversal, `.git`, virtual environments, caches, `node_modules`,
root escape, and existing symlink escape are rejected. Export paths are fixed by
policy; users cannot provide one. CLI and JSON expose only normalized relative
paths.

## 11. Resource bounds

Compiled caps limit policy bytes, trace bytes, trace backups, scanned files,
records, line/record bytes, collection items, identifier length, index bytes,
report UTF-8 bytes, retained reports, and files inspected for cleanup. Files
above a validated bound are skipped instead of loaded without limit.

## 12. Backward compatibility

The following remain supported:

```text
/doctor
/doctor trace latest
VEGA_EXECUTION_TRACE
append_trace(...)
load_latest_trace(...)
serialize_trace(...)
format_trace_summary(...)
```

Existing valid v2.10 JSONL records have the same allowlisted schema and remain
readable. Public constants remain exported. Older project roots without the new
policy retain safe trace-read defaults; export still requires a real policy.

## 13. Failure semantics

Diagnostics are fail closed. Invalid configuration yields no partial policy.
Broken optional subsystems become fixed degraded/unavailable sections while a
safe report can still be built. A fatal production snapshot marks the report
blocked but grants no capability. Corrupt traces are skipped. Serialization and
export failure leave no partial JSON. Retention failure preserves a completed new
report. No diagnostic failure invokes or retries a tool or changes a response.

## 14. Test matrix

Coverage includes strict policy schema/types/paths/caps/duplicates/symlinks;
frozen models and exact fields; deterministic UTF-8 bounded serialization;
secret sentinels; healthy/degraded/blocked subsystem state; opt-in and empty
traces; corruption; exact-threshold rotation and `.1/.2/.3` order; active/backup
latest lookup; record/file scan caps; v2.10 compatibility; aggregates; explicit
export, safe filename, atomicity, size failure, retention, and unknown-file
preservation; all doctor commands; Git-ignore state; identity and release policy.

## 15. Acceptance criteria

Acceptance requires validated policy, immutable safe reports, explicit atomic
export, bounded report retention, three-backup trace rotation, bounded scans,
v2.10 trace compatibility, exact doctor commands, absence of sentinel leakage,
path confinement, ignored generated state, synchronized v2.11 identity and docs,
focused/full pytest, compileall, identity, policy consistency, smoke, Release
Manager, and whitespace validation.

## 16. Known limitations

* Trace persistence is opt-in.
* Reports and diagnostics are local-only; remote telemetry is absent.
* The trace lock coordinates only threads in one process and is not interprocess
  locking.
* The report builder does not launch Ollama to probe installed models.
* No automatic publishing is provided.

## 17. Migration impact

No data migration is required. Add the tracked diagnostics policy and retain
generated state outside Git. Valid v2.10 traces continue to load. Integrations
may adopt new report/status APIs incrementally; existing trace calls and doctor
latest output remain compatible.

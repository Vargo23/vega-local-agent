# Migrating VEGA from v2.13.0 to v3.0.0

This is the project migration guide for the `v3.0.0` release. The official
upstream history available when this guide was prepared ended at `v2.13.0`; the
rules below therefore describe only changes present in the VEGA source tree and
do not attribute behavior to an external package or SDK.

## Before upgrading

1. Finish or inspect any active controlled workflow with `/workflow status`.
2. Keep a copy of local ignored state only if your normal backup policy requires
   it. v3 does not rewrite that state.
3. Record the selected model with `/model` and run `/doctor state status`.
4. Run the v2.13 test/release checks from a clean checkout when possible.

## Required application changes

No configuration or stored-data conversion is required. Existing CLI commands,
model profiles, permissions, plugins, workflow state, checkpoints, traces, and
diagnostic reports retain their current contracts.

Update only integrations that depend on exact terminal text:

* replace assertions for the framed ASCII banner with
  `VEGA / OPERATOR CONSOLE`, version, workspace, model, and readiness checks;
* replace the old two-line prompt expectation with `vega ›` or `vega >`;
* do not expect Internet, network, safety, or session-log paths at startup;
* treat progress output as ephemeral presentation, not a machine protocol;
* update exact output snapshots for the elapsed timer and final usage summary;
* allow nullable token counts in new `REQUEST_METRICS` session-log sections.

Python callers may continue using the pre-v3 positional arguments of
`render_startup_screen(...)` and `VegaStatus(model, internet, version)`.
Internet and log-path arguments are accepted for source compatibility but are
not displayed.

Code importing the internal `core.agent_runtime.banner()` or
`fallback_banner()` helpers must switch to `ui.startup_screen` or
`scripts.vega_banner.render_banner()`. Do not launch
`scripts/vega_banner.py` directly; `scripts/vega.py`, `vega.cmd`, and the
provided batch/PowerShell wrappers are the supported entrypoints.

## Installation and verification

```powershell
python -m pip install -r requirements.txt
python -m compileall -q core tools permissions plugins domains workflows planner memory rag review scripts tests ui web_demo
python -m pytest -q -rs -p no:cacheprovider --basetemp .tmp/pytest-release --tb=short
python scripts/check_identity.py
python scripts/check_policy_consistency.py
python scripts/smoke_test.py
python scripts/check_repository_hygiene.py
python scripts/run_release_check.py
```

Then launch `vega.cmd` or `python scripts/vega.py` and verify the Operator
Console, one prompt, model/workspace labels, live progress, confirmation waits,
and `/exit` behavior. Send two sequential chat requests and confirm the second
timer starts from zero. For a successful Ollama response with usage, confirm
the final total equals input plus output. Also exercise one cancellation or
timeout and confirm the timer stops and missing usage is reported as
unavailable rather than zero. Redirect output once and confirm it contains
ordinary sequential lines without TTY redraw controls.

## Rollback

No reverse data migration is needed. Stop VEGA, preserve any desired ignored
local state, and start a clean checkout of tag `v2.13.0`. Do not copy v3 source
files into the older checkout. Because v3 does not change persisted schemas,
valid v2.13-compatible state remains readable; keep backups and use the older
release's explicit repair commands rather than manually editing state files.
The additive v3 `REQUEST_METRICS` sections require no reverse conversion and
may be ignored by v2.13 because session logs are diagnostic text rather than
loaded application state.

# VEGA v3.0 Architecture

Maintained release: `v3.0.0`

## Scope

v3.0 changes the terminal presentation and adds request-local progress
observability. It does not replace orchestration, `PlanExecutor`,
`ToolExecutor`, production policy, model selection, controlled workflows,
plugins, permissions, diagnostics, or release publication rules.

```text
scripts/vega.py
    -> core/agent_runtime.py
        -> ui/startup_screen.py
        -> ui/terminal_prompt.py
        -> core/request_metrics.py (request-local clock and usage)
        -> core/ollama_client.py (exact server usage)
        -> core/contextual_runtime.py
            -> core/execution_progress.py
            -> core/plan_executor.py
                -> ToolExecutor.execute() exactly once per planned step
            -> ui/terminal_progress.py (observer only)
        -> ui/request_summary.py
```

## Operator Console contract

The runtime supplies four values to the startup renderer:

* `scripts.version.VERSION`, the canonical release identity;
* `ExecutionContext.project_root.name`, the active workspace label;
* `ExecutionContext.model`, the selected model;
* `ProductionRuntimeBootstrap.status`, the readiness state.

Terminal width is detected at render time with a safe 80-column fallback. The
version remains on the title line only when both values fit, the separator never
exceeds the width, and long workspace/model values are truncated. Unicode
`◇`, `─`, and `›` have `*`, `-`, and `>` fallbacks. ANSI is capability-gated;
plain output contains no control sequences.

The startup renderer never reads input. `ui/terminal_prompt.py` supplies the
single prompt used by the existing runtime loop.

## Progress contract

`ExecutionProgressEvent` is frozen, slotted, bounded, and contains only stage,
step counters, display titles, plan titles, and optional elapsed time. Callback
delivery is optional and fail-soft. A callback failure cannot alter a plan,
grant permission, execute a tool, retry a tool, change its result, or persist
new history.

Progress bars are shown only after a validated plan establishes the real step
count. Interactive terminals may redraw one line; redirected output is a
sequence of ordinary lines.

The renderer's daemon timer thread can only read elapsed time and redraw the
active TTY line. It cannot call the model, execute tools, mutate a plan, or
persist data. Every terminal event and `close()` stop the thread; non-interactive
renderers do not start it.

## Request metrics contract

`RequestMetrics` is created once per chat request and is never shared through
global state. It uses a monotonic clock for duration and UTC wall-clock values
for session history. Phase durations cover preparation, model wait, response
processing, tools, and saving; phases remain diagnostic and are not rendered as
internal implementation detail.

Ollama usage is accepted only when both `prompt_eval_count` and `eval_count`
are non-negative integers returned by the server. Total usage is their exact
sum. Streaming usage is read from the final `done` chunk. If any model call in
a request lacks complete usage, the request usage remains unavailable rather
than mixing exact and guessed values.

Success, failure, policy block, cancellation, and timeout stop one request's
tracker independently. A structured `REQUEST_METRICS` session-log entry stores
UTC start/end time, duration, status, nullable input/output/total counts, and
phase durations. It does not add request or response content.

## Compatibility and breaking changes

The intentional breaking boundary is the text UI. Exact snapshots/parsers for
the v2 startup banner or two-line prompt must migrate. Slash commands, runtime
execution results, tool call counts, permission decisions, plugin API version,
configuration schemas, and persisted state formats are unchanged.

The legacy `call_ollama_chat(...) -> (ok, content)` contract remains available.
The detailed `request_ollama_chat(...)` result adds status and exact nullable
usage without breaking existing callers.

The legacy positional parameters of `render_startup_screen(...)` remain
accepted, although Internet and log-path values are no longer rendered.
`VegaStatus(model, internet, version)` remains constructible and delegates to
the canonical UI renderer.

The internal `core.agent_runtime.banner()` and `fallback_banner()` loaders are
removed so the old presentation cannot survive as a fallback path. Direct
execution of `scripts/vega_banner.py` is also removed; `scripts/vega.py` is the
only CLI entrypoint. Import callers may use the compatibility `render_banner()`
wrapper, which delegates to `ui.startup_screen`.

## Data migration

None. v3 introduces no schema-version increment and performs no startup repair
or conversion. Existing state continues through the v2.13 readers and explicit
repair paths. Session logs gain additive `REQUEST_METRICS` sections; older logs
remain readable. Invalid, oversized, ambiguous, or unsafe state still fails
closed.

## Dependencies and infrastructure

No dependency or Python support change is required. The project has no lock
file, package build manifest, database migration framework, Dockerfile, or
deployment manifest. Existing GitHub Actions continue to test Python 3.12–3.14
on Windows and Linux and retain read-only permissions.

## Validation boundary

Release validation covers startup content, narrow widths, long runtime values,
ANSI/no-color output, Unicode/ASCII fallback, legacy Python call shapes,
progress event validation, callback failure isolation, exact tool call counts,
timer lifecycle, duration/locale formatting, exact and missing usage, streaming,
cancellation, timeout, reset, parallel metric isolation,
full tests, compileall, identity, policy consistency, smoke, documentation,
repository hygiene, and the built-in Release Manager.
